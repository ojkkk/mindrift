## 二、当前核心问题诊断

### 问题 1：平台单一依赖（高风险）

**现状：** Mindrift 100% 绑定 `~/.codex/sessions/` 这个目录和 OpenAI Codex 的 JSONL 格式。

**风险：**
- Codex 改版日志格式 → 解析器失效
- OpenAI 停更 Codex 桌面端 → 整个项目失去意义
- 用户迁移到其他 AI Coding 工具 → 受众流失

**这不是假设，而是已经在发生：** 2026 年市场上 Claude Code、Gemini CLI、Cursor 的 Agent 模式用户量均已超过 Codex。

### 问题 2：可视化天花板

**现状：** 图表全部用纯 SVG 手写，没有引入图表库。

**后果：**
- 复杂图表（热力图、散点图、桑基图）实现困难
- 维护难度高，新贡献者上手门槛大
- 相比竞品在视觉表现力上有明显短板

### 问题 3：无协作、无分享

**现状：** 100% 本地，无法把 session 分析结果分享给他人。

**后果：**
- 团队场景完全缺失（技术 Leader 无法看团队整体 AI 使用情况）
- 无法形成口碑传播（"发给同事看看"这个动作做不到）
- 企业版潜力无法释放

### 问题 4：开发计划缺少破局性功能

**现状 TODO：** 多 Session 对比、趋势图、数据导出、自定义告警、插件系统。

**问题：** 这些都是锦上添花，不是护城河。竞品可以在两周内复制这些功能。

## 三、时代背景与机会窗口

### 2026 年的 AI Agent 市场格局

```
2024年         2025年         2026年（当前）
  │              │               │
Copilot主导    Agent爆发期     可观测性缺口凸显
单步代码补全    多步自主任务    "AI在干嘛"成为真实痛点
                               Token账单膨胀
                               企业合规需求涌现
```

**三个正在发生的趋势，Mindrift 正好处于交叉点：**

1. **AI Agent 普及** → 用户从被动接受建议变成主动委托任务，"可观测性"需求从可选变为必须
2. **Token 成本意识觉醒** → 个人月账单数百美元，企业月账单数万美元，精细化管理刚需涌现
3. **AI 工具碎片化** → 每个团队成员用不同的 AI 工具，管理者需要统一视图

Mindrift 的"本地日志读取"架构，恰好是应对这三个趋势最低成本的切入方式。

## 四、发展路线图（三阶段）

### 阶段 A：突围扩圈（优先级最高，1-2 个月）

**目标：** 从"Codex 专属工具"变成"AI Coding 工具通用监控台"

#### A1：接入 Claude Code（最高优先级）

Claude Code 在 `~/.claude/projects/` 下存储会话日志，格式为 JSONL，与 Codex 高度相似。

```
实现成本：新增一个 claudeParser.js，约 200-300 行代码
预期收益：受众扩大 3-5 倍，README 从"Codex 专属"升级为"多平台支持"
```

**Claude Code 日志路径：**
```
~/.claude/projects/<project-hash>/
  ├── <session-uuid>.jsonl
  └── ...
```

关键事件类型（需要适配）：
- `user` / `assistant` 消息对
- `tool_use` / `tool_result` 工具调用
- `usage` token 统计
- `model` 字段（sonnet/opus/haiku）

#### A2：接入 Cursor Agent 模式日志

Cursor 的 Agent 模式在 `~/.cursor-tutor/` 或工作区 `.cursor/` 下有部分日志。优先级低于 Claude Code，但用户基数大。

#### A3：多平台统一解析层架构

重构解析器，从单文件变成插件化架构：

```
parsers/
  ├── index.js          ← 自动检测 + 路由
  ├── codex.js          ← 现有逻辑迁移
  ├── claude-code.js    ← 新增
  ├── cursor.js         ← 新增（可选）
  └── base.js           ← 统一输出格式定义
```

### 阶段 B：商业化布局（2-4 个月）

**目标：** 建立可持续的收入模式，从个人工具升级为企业工具

#### B1：团队版功能（核心变现点）

**企业真实痛点：** 技术 Leader 不知道团队每个人在 AI 工具上花了多少、花在什么上、效率怎么样。

```
团队版功能清单：
  - 多人数据汇聚（各成员的本地 agent 上报到团队服务）
  - 团队 Token 消耗仪表盘（按人/按项目/按时段）
  - 异常用量告警（某人 token 突然暴增，可能是 agent 失控）
  - 成本归因报告（哪个项目最烧钱）
  - 导出 CSV，对接企业财务系统
```

**定价建议：**
- 个人版：永久免费（保持开源）
- 团队版：$15/人/月（5 人起）
- 企业版：$499/月（无限人数 + 私有部署）

#### B2：分享与协作

**轻量实现方案：**

```
功能：一键生成"Session 摘要分享卡"
  - 图片形式（PNG）或网页链接
  - 包含：任务描述、总 token 消耗、工具调用次数、耗时
  - 可发到飞书/Slack/微信
  
技术方案：html-to-image 库，纯本地生成，不上传数据
```

#### B3：Webhook / 集成

支持将统计数据推送到外部系统：

```javascript
// 配置示例（mindrift.config.json）
{
  "webhooks": [
    {
      "event": "session_end",
      "url": "https://hooks.feishu.cn/xxx",
      "payload": {
        "total_tokens": "{{session.totalTokens}}",
        "task_summary": "{{session.name}}"
      }
    }
  ]
}
```

### 阶段 C：建立护城河（4-6 个月）

**目标：** 从"功能产品"变成"平台"，让竞品难以追赶

#### C1：AI 使用效率评分体系（核心护城河）

**核心洞察：** Token 消耗量不等于工作效率，需要建立"AI 使用质量"的评价体系。

```
效率指数 = f(任务完成率, Token利用率, 工具调用精准度, 重试率)

具体指标：
  - 每个 token 完成多少有效工作（token ROI）
  - 工具调用成功率 vs 错误率
  - context 压缩前浪费了多少 token
  - 计划执行率（update_plan 步骤完成比例）
```

**这是壁垒：** 这套评分体系需要大量真实数据校准，积累越多越准，新进者无法快速复制。

#### C2：跨 Session 模式挖掘

**挖掘有价值的 pattern：**

```
自动识别：
  - 哪类任务最容易让 Agent 失控（token 暴增）
  - 哪些工具组合最高效
  - 用户的"AI 使用习惯画像"（重度工具调用 vs 重度对话型）
  - 异常 Session 特征提取（超长、频繁中断、大量错误）
```

#### C3：MCP 服务器模式

将 Mindrift 暴露为一个 MCP（Model Context Protocol）服务器，让 AI Agent 自己能查询自己的历史用量和效率数据。

```
场景：用户对 Claude Code 说"告诉我本周我在哪个项目上花了最多 token"
Claude Code → MCP 调用 Mindrift → 返回结构化统计数据

这是一个"AI 自我观测"的闭环，极具前瞻性
```

## 五、技术债偿还建议

### 图表库迁移（建议 Phase A 内完成）

**当前问题：** 纯 SVG 手写图表，复杂图表无法实现。

**建议方案：** 引入 Recharts（React 原生，Tailwind 友好，MIT 协议）

```bash
npm install recharts
```

**迁移优先级：**
1. `TurnTokenChart.jsx` → Recharts `<BarChart>` （影响最大）
2. `TokenDonut.jsx` → Recharts `<PieChart>` （代码量减少 70%）
3. 新增趋势折线图 → `<LineChart>` （当前无法实现）

### 配置文件系统

**当前：** 端口、路径都是硬编码在 `server/index.js`。

**建议：** 新增 `mindrift.config.json`

```json
{
  "port": 3344,
  "sources": [
    { "type": "codex", "path": "~/.codex/sessions" },
    { "type": "claude-code", "path": "~/.claude/projects" }
  ],
  "theme": "dark",
  "webhooks": []
}
```

## 六、竞争格局分析

| 竞品 | 定位 | 与 Mindrift 的差异 |
|------|------|-------------------|
| OpenAI 官方用量页 | Token 账单统计 | 无 Agent 行为细节，无本地可视化 |
| Cursor Analytics | Cursor 专属 | 闭源，数据上云，无自托管 |
| LangSmith | LangChain 生态可观测性 | 需要代码侵入，面向开发者 |
| Helicone | API 代理统计 | 需要改 API endpoint，有隐私顾虑 |
| **Mindrift** | **本地零侵入多平台** | **唯一不需要任何代码修改的方案** |

**结论：** Mindrift 的"零侵入"壁垒是真实的，竞品要复制这个架构，需要为每个平台逆向日志格式，成本不低。
